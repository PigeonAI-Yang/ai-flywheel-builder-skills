# 记忆更新规则

## 写入长期记忆的条件

只有会影响未来判断的信息才写入长期记忆。

经验收口中，至少满足以下一项才允许升级为长期经验：

- 问题重复出现
- 用户明确裁定
- 造成流程中断
- 影响下一次判断
- 暴露模板或模块缺陷
- 属于平台或工具限制

## 必须写入

- 用户身份变化
- 目标受众变化
- 公开边界变化
- 内容主线变化
- 反复出现的内容种子
- 已验证有效的实践模式
- 已验证无效的实践模式
- 反馈中反复出现的用户卡点
- 反复出现的模块失误
- 已确认的平台或工具限制
- 会影响后续判断的避坑规则
- 已确认的对外表达顺序和公开边界

## 不写入

- 一次性情绪
- 未验证想法
- 与当前飞轮无关的旁支
- 临时措辞
- 没有证据的一次性判断
- 未造成影响的偶发错误
- 用户私聊、情绪表达和调试过程本身；它们只能作为内部规则修正证据，不能原样进入公开稿

## 对外内容避坑规则

- 内容飞轮的公开稿先讲目标读者的问题和判断价值，再讲技术细节。
- 技术细节、状态机、中控、索引、验收脚本、后台文件只能在后半段作为可信度背书。
- 用户反馈可以改变产品规则，但不能原样进入对外内容。
- 公开稿要围绕真实实践 -> 判断 -> 内容 -> 产物 -> 反馈 -> 下一轮实践展开。
- 对 AI Flywheel Builder 这类产品，最值得讲的是它如何判断一个具体项目值不值得讲，而不是它生成了哪些后台交付物。
- 公开稿必须使用目标读者会说的人话；自造术语、内部口径、项目黑话和非行业通用词必须删除、翻译或后置解释。
- `主交付物`、`经验收口`、`状态流转`、`模块路由`、`run_id`、`索引`、`验收脚本` 这类词默认不进公开稿前半段。
- 公开稿必须尽量包含真实实践里的困难、尝试、失败、解决方式和关键转折；缺少这些细节时，要引导用户补充并继续修订同一篇稿。
- 交付稿不是一次性终稿；内容飞轮应把用户补充、修正和反馈接回原稿，直到能发布或明确暂缓。
- 产品、skill、workflow 类公开稿默认使用产品发布长文协议：真实痛点 -> 问题反转 -> 产品出现 -> 判断机制 -> 实践摩擦 -> 多 skill 系统背书 -> 可传播结论。
- 写 AI Flywheel Builder 时，必须表达为母 skill + 多个子 skill 协作的内容创作飞轮，不能降级成 prompt 或单个写作 skill。
- 可读的产品介绍不等于可发布内容；没有尖锐问题、具体案例入口、失败诊断、方法增量和可执行动作，只能标记为可修订框架稿。
- 产品、skill、workflow 类稿件必须先交内容框架，让用户确认“是不是讲这些事、角度是否对”，再补采真实细节并修订同一篇稿。
- `content-flywheel-run.html` 和 `content-flywheel-run.md` 只能放外部读者能看的内容、必要补采问题和修订入口；run、索引、验收、旧输出作废、后台记录和已更新文件只能放后台文件或最终回复的路径清单。
- 用户提交产品目录、skill 目录、workflow 目录、手册目录或工具项目时，目录本身就是高密度材料。AI 必须先提取产品事实层，不得把产品是什么、解决什么痛点、有什么功能、怎么使用等目录内可得信息重新问用户；用户采访只补创作者经历层。
- 扫描产品目录时必须站在外部读者视角提取：这跟我有什么关系、使用前后差异、最小上手路径、可见样例、可信证据、关键设计取舍、同类对比、边界限制、目标用户分层、可复用部分、可传播观点和反馈入口。
- 公开主交付稿不能出现“要讲的判断”“这一节要避免”“不要写成”“讲清楚”“要展示”这类作者批注或模型指令；这些只能放在后台拆解和 skill 规则里。主交付稿只能放读者能直接看的正文、框架、必要补采问题和下一版修改方向。
- 主交付物打开后必须让用户马上知道：推荐怎么讲、讲给谁、为什么这么讲、现有材料能支撑什么、还缺哪几句真实经历、用户现在只需要判断什么。交付物清单、路径表、验收结果、已更新文件、后台自检表和路由字段只放最终回复或后台文件，不进主交付稿正文。
- 写 skill、workflow、开发工具、技术产品时，不要把“AI 创作者”和“skill/workflow 开发者”粗暴二选一。优先找交集人群，例如“正在做 AI skill / workflow 的实践型创作者”。传播顺序可以前半段讲痛点、后半段讲系统，但目标读者不能把用户本人排除掉。
- 主交付物末尾必须给出可直接回复的反馈格式，明确用户现在只需要反馈哪 1-3 件事；用户说“看完了”以后，下一次回复必须先引导反馈，而不是继续解释文件改动。
- 对可修订稿、决策稿、框架稿、首发稿或产品发布文，最终对话回复必须在交付物清单前直接问 1-3 个具体反馈问题，并给出可复制回复格式；不能只依赖 HTML/Markdown 文件里的反馈入口。
- 用户确认内容方向后，不能跳过补采直接扩成发布稿。方向确认只说明框架成立；下一步必须补使用前后对比、关键转折和公开边界，补完后再扩写。
