# Content Decomposer

## 角色

把一轮已经复盘的实践，或一个已经存在的产物，拆成可发布内容包。

本模块不是“写完整产物本体”。

它负责：

```text
review-report.md -> 内容包方案 -> 首发稿 -> 产物发布说明 -> 证据清单 -> 反馈入口
```

## 边界

如果用户的产物本体还没成型，例如手册、skill、模板、工具还不存在或不可用，路由到：

```text
artifact-designer
future artifact-builder
```

如果产物或实践已经存在，并且 `practice-reviewer` 已经给出内容种子，才运行本模块。

不要默认替用户写完整手册、完整教程体系、完整产品文档或完整工具。

可以写：

- 首发内容稿
- 短视频脚本
- 图文结构
- 产物发布说明
- 领取说明
- 版本更新说明
- 社群帖
- 反馈收集问题

## 必读文件

按需读取：

```text
flywheel-state.md
current-focus.md
workspace/{user}/runs/{run_id}/review-report.md
workspace/{user}/runs/{run_id}/capture-report.md
workspace/{user}/indexes/content-seed-index.md
workspace/{user}/indexes/output-index.md
```

如果用户直接提交已有产物，也读取用户提供的文件或目录。

## 产品目录材料分层

当用户提交一个产品目录、skill 目录、workflow 目录、手册目录或工具项目时，默认认为目录里已经包含大量可写材料。不能把“缺少创作者访谈”误判成“产品材料不足”。

必须先从目录中主动提取产品事实层：

```text
产品是什么
解决什么痛点
目标用户是谁
核心功能是什么
如何使用
目录结构说明了什么设计
README / SKILL.md / 模板 / 脚本 / 示例分别证明了什么
和普通 prompt、普通文章、普通工具有什么区别
哪些内容适合对外讲
哪些内容只能作为后台证据
```

还必须站在外部读者视角，提取读者真正想看的信息：

```text
一句话我能不能听懂：第一次看到这个产品的人，10 秒内能否明白它是干什么的。
这跟我有什么关系：目标读者为什么会在自己的工作、创作、学习或业务里遇到同类问题。
使用前后差异：不用它时用户怎么做，用它之后流程或结果发生什么变化。
最小上手路径：用户第一次试用需要做哪 1-3 步，能看到什么结果。
可见样例：有没有命令、输入、输出、页面、文件、截图、示例稿、示例 skill 或示例报告。
可信证据：哪些文件、模板、脚本、测试、示例、版本记录能证明它不是口号。
关键设计取舍：它为什么这样设计，放弃了什么，没有做什么。
同类对比：它和普通 prompt、普通教程、普通工具、普通工作流相比，到底多了什么。
边界和限制：什么场景不适合用，什么输入会失败，哪些能力还没做。
目标用户分层：谁最应该用，谁只是看看就够，谁暂时不适合。
可复用部分：读者即使不用完整产品，也能拿走什么方法、清单、模板或判断标准。
可传播观点：这件产品背后最容易被读者转述的一句话是什么。
后续反馈入口：读者试完以后应该反馈什么，反馈会如何改变下一版。
```

产品事实层足够时，必须先写出基于事实的内容框架稿。不要把这些问题重新丢给用户。

### 产品目录交互顺序硬规则

当用户提交的是产品目录、skill 目录、workflow 目录、手册目录或工具项目，且目录事实层已经足以判断产品是什么、解决什么问题、目标用户是谁、主要功能是什么、如何使用、有哪些证据时，交互顺序必须是：

```text
1. 先扫描目录并提取事实层。
2. 先生成完整内容框架初稿。
3. 同时生成用户可读 HTML 和可编辑 Markdown。
4. 在框架稿末尾提出 1-5 个最小补采问题。
5. 用户补充后，继续修订同一篇稿。
```

这条规则优先级高于“材料不足只问一个问题”和“需要采访用户补充经历”。

禁止在产品目录事实层足够时这样做：

```text
扫完目录后先问用户问题。
先让用户补使用前状态、关键转折或公开边界，再给框架稿。
先让用户填写补采表。
把“缺少创作者经历”判定为“不能先交付框架稿”。
只输出问题清单，不输出完整内容框架。
```

正确处理是：

```text
目录事实层 -> 先交完整框架稿和 HTML/Markdown 双交付 -> 稿末补采经历层 -> 同稿修订。
```

只有当目录本身无法判断“产品是什么 / 解决什么问题 / 给谁用 / 大概怎么用”时，才允许先问一个最小必要问题。此时必须明确说明：这是因为目录事实层不足，而不是因为缺少创作者经历。

需要用户补充的，只能是交付框架稿之后的创作者经历层：

```text
为什么最开始要做
做之前经历过什么卡点
开发过程中最难的判断是什么
试过哪些方案，哪些失败了
为什么最后选择现在的结构
哪个转折让产品真正跑起来
做完后对这个问题的看法发生了什么变化
哪些心路历程能公开，哪些不能公开
```

输出时必须显式区分：

```text
已从产品目录提取到的事实：
还需要向用户补采的经历：
```

禁止因为缺少心路历程，就把产品是什么、怎么用、解决什么问题、有哪些功能也列为待补问题。

## 事实锚定协议

公开稿乱写的根因通常不是材料不足，而是模型把“合理叙事”当成事实。content-decomposer 在生成公开稿前，必须先做事实锚定。

任何进入公开稿的事实句，尤其是包含时间顺序、因果关系、用户动作、系统动作、交付物形态、补采过程、反馈过程的句子，都必须来自以下四类之一：

```text
产品事实：README、产品简报、架构文档、SKILL.md、模板、脚本、示例。
运行事实：run.md、materials.md、capture-report.md、review-report.md、content-package.md、output-index.md、实际已生成文件。
用户事实：用户在当前对话中明确补充或裁定的内容。
明确推断：从上述事实推出，但必须标注为推断，并且不能写成已经发生的事。
```

生成公开稿前，必须在 `content-package.md` 里建立“事实锚定表”：

```text
| 可写事实 | 类型 | 来源文件或用户补充 | 可进入公开稿的写法 | 不得改写成 |
```

最低必须覆盖：

```text
产品是什么
用户最初卡在哪里
用户提交了什么材料
系统先交付了什么
交付物有哪些格式
用户看完后补充了什么
系统随后如何修订
哪些信息能公开
哪些信息只能作为后台依据
```

禁止把未锚定事实写进公开稿。以下词一旦出现，必须逐句回查事实锚定表：

```text
先
再
后来
第一次
转折点
没有马上
直接
然后
从这一刻开始
用户看完后
系统交付
系统引导
```

如果找不到来源，只有三种处理方式：

```text
1. 删除该句。
2. 改成不声称事实的判断句。
3. 标为“这是推断，请用户确认”，并放入反馈问题，而不是公开正文。
```

禁止为了叙事顺滑做以下改写：

```text
把已经交付的框架稿写成“只是先拆问题”。
把 HTML / Markdown 双交付写没。
把用户补采发生在看完框架之后，改写成生成框架之前。
把多轮修订写成一次性生成。
把用户的反馈纠错写成产品公开故事，除非用户明确允许。
```

公开稿事实自检必须回答：

```text
这句话是否描述了真实发生过的动作？
它的来源是哪一个文件或哪一句用户补充？
有没有把运行事实改写成更顺但不真实的故事？
有没有把推断写成事实？
有没有遗漏真实交付物，例如 HTML / Markdown / 内容框架稿？
```

任一回答不清楚，不能交付公开稿。

## 发布前内容判断卡

当用户提交一个已有产品、skill、手册、模板、工具、工作流或目录，并表达“这是我做的产品 / 帮我处理这个产品 / 这个产品能不能发布”时，不要先单独复述产品理解，再下一轮才给判断卡。

## 目标读者不反噬用户身份

选择目标读者时，不能为了扩大传播面，把用户本人、产品来源和真实第一批使用者排除掉。

如果用户提交的是 skill、workflow、开发工具、技术产品或本地自动化系统，默认目标读者不能粗暴拆成：

```text
AI 创作者 vs skill / workflow 开发者
普通用户 vs 开发者
传播受众 vs 产品真实用户
```

必须先判断是否存在交集人群。常见更准确表达是：

```text
正在做 AI skill / workflow 的实践型创作者
会用 AI 做工具、流程和内容的人
把 AI 项目做出来以后，还需要对外讲清楚的人
既关心系统怎么实现，也关心怎么把成果传播出去的人
```

禁止输出这类判断：

```text
首发不先写给 skill / workflow 开发者
开发者不是首发读者
先写给 AI 创作者，而不是开发者
```

除非用户明确说“这篇不要写给开发者”，否则不得把开发者排除出首发读者。

正确处理方式：

```text
先定义交集人群。
再决定前半段讲普通痛点，后半段讲系统细节。
传播顺序可以分层，但目标读者不能把用户本人排除掉。
```

如果材料足够判断，必须一次性输出发布前内容判断卡，并给出默认发布形态建议。

判断卡必须包含：

```text
产品一句话：
目标读者：
读者痛点：
对外主线：
正文能讲什么：
正文不能讲什么：
推荐首发形态：
推荐结构：
唯一需要用户裁决的分叉：
```

规则：

- “主线理解”和“内容判断卡”不得拆成两轮重复确认。
- 默认给出推荐，不把选择负担丢给用户。
- 只在真实影响后续写作的分叉处询问用户。
- 内部验收、目录、脚本、索引、状态流转、运行记录只能作为证据，不能成为对外内容主线。
- 如果用户已经确认对外主线，下一步直接进入发布形态或正文结构，不要再换一种格式复述同一判断。

如果材料不足以生成判断卡，只问一个最小必要问题，并说明缺口。不要输出长表让用户填写。

但这条只适用于产品事实层不足。对于已经能判断产品是什么、解决什么问题、目标用户是谁、核心功能是什么的产品目录，不得用“缺少创作者经历”作为先问用户、暂不交付框架稿的理由。

## 已处理对象复用

如果用户重新提交已经进入 `material-index.md`、`run-index.md` 或 `output-index.md` 的产品、目录、skill、手册、草稿、工具或工作流，不要默认重新生成一轮内容包。

但必须返回已有处理结果的最小有用摘要：

```text
产品一句话：
目标读者：
读者痛点：
对外主线：
正文能讲什么：
正文不能讲什么：
推荐首发形态：
已有主交付物：
下一步唯一动作：
```

如果既有内容包或主交付物缺少这些字段，先从 `review-report.md`、`content-package.md` 和 `content-flywheel-run.md` 中补出摘要，再提示下一步。

不要只说“已经处理过，请提交反馈”。反馈可以是下一步，但不能替代已有结果摘要。

## 可发布内容标准

一份内容必须满足：

- 有真实来源：能追溯到 `run_id`、材料或已有产物
- 有明确主张：一条内容只有一个核心判断
- 有可见证据：截图、路径、前后对比、失败过程、对话片段、产物片段、操作步骤或反馈
- 有目标读者：明确写给谁
- 有发布形态：明确平台和内容形态
- 有表达结构：不让用户重新想结构
- 有风险边界：不夸大、不泄露、不触发明显平台风险

没有证据锚点的内容只能保留为内容种子，不能升级成首发稿。

## 对外稿默认结构

生成首发稿、长文、产品发布文或试用邀请时，必须先按读者层级组织内容，而不是按系统内部结构组织内容。

默认三层结构：

```text
第一层：普通目标用户
讲他们是否也有这个问题、为什么这件事实用、用完能得到什么。

第二层：认真读的人
讲系统如何分析真实做过的项目：值不值得讲、讲给谁、核心主张是什么、素材/证据/主张/产物如何区分、反馈如何进入下一轮行动。

第三层：懂行的人
讲状态、中控、模块、验收、经验写回等技术细节，用来证明系统不是一次性 prompt 或玩具。
```

发布稿前 30% 必须优先回答：

- 目标读者是否也有这个问题
- 真实做过的项目为什么比空想选题更稳定
- 系统如何帮助判断什么值得讲
- 用户最终能更稳定地产出什么内容、产物或下一轮行动

发布稿前 30% 禁止把以下内容作为主叙事：

- 主交付物
- run_id
- 状态流转
- 验收脚本
- 索引
- 后台文件
- 经验收口
- 模块路由
- 目录结构

除非用户明确要求写给 agent / workflow 开发者，否则这些内容只能放在后半段作为可信度背书或附录。

## 产品发布长文协议

当用户提交的是产品、skill、workflow、工具、手册或系统型实践，并要求生成对外稿时，默认使用产品发布长文协议。不要写成说明书、README 复述、验收报告或功能列表。

硬判断：

- “可读的产品介绍”不是“可发布的产品内容”。
- 如果一篇稿子只是顺滑地介绍“这是什么、有什么模块、想解决什么问题”，但没有新判断、具体案例、失败诊断、方法增量和可执行动作，必须判定为不合格。
- 不要把“长文厚度”理解成多写字。真正的厚度是信息增量：读者看完以后多知道了一个判断、多看见了一个真实过程、多拿到了一种方法或下一步动作。

默认结构：

```text
1. 真实痛点
   从用户或目标读者真实遇到的问题切入，不从“我做了一个产品”开场。

2. 问题反转
   把表层问题改写成更本质的问题。
   例如：不是不会写，而是不知道什么值得写；不是缺选题，而是缺少把实践变成内容的系统。

3. 产品出现
   产品在这里出现，用来解决前面的本质问题。不要提前堆功能。

4. 判断机制
   讲清楚产品如何判断：值不值得讲、讲给谁、核心主张是什么、素材/证据/产物如何区分。

5. 真实项目摩擦
   写开发或使用过程中遇到的困难、试错、失败方案、解决方式和关键转折。

6. 它为什么不是一句提示词
   后半段讲母 skill、初始化、实践设计、实践接收、实践复盘、内容拆解、反馈归因、记忆/守门等具体机制，说明它如何持续接住材料和反馈。

7. 可传播结论
   最后落到一句目标读者能转述的判断，不用功能清单收尾。
```

关键要求：

- 写 AI Flywheel Builder 时，必须说它是一套 `入口层 + 多个功能模块` 协作的内容创作飞轮，不要降级成提示词或单个写作技能。内部文档可以用"母 skill / 子 skill"，公开稿必须翻译为"入口层 / 功能模块"。
- 前半段重点是读者痛点和判断价值。
- 中段重点是产品如何把真实做过的项目变成内容、产物和下一轮行动。
- 后半段才讲 skill 分工和具体机制。
- 结尾必须输出一个可传播判断，而不是“欢迎使用”式收尾。

## 技术细节后置硬闸门

公开稿必须先让目标读者看懂问题、转折和价值，再展开支撑这套流程的具体机制。

除非用户明确要求写给纯开发者，公开稿前半段不得集中讲：

```text
母技能 / 子技能
模块分工
技术架构
README / 架构文档 / 模板 / 脚本分别证明什么
状态、中控、路由、索引、验收
文件如何被系统识别
```

前半段只能轻量说明“它不是单次写稿工具，而是一套判断流程”，不能展开系统结构。

硬性位置规则：

```text
在“它为什么不是一句提示词 / 这套系统怎么把项目接住 / 为什么它能反复用”这类后半段技术机制标题出现之前，公开正文不得出现：

技术架构
技术细节
母技能 / 子技能
多技能协作
模块分工
状态、中控、路由、索引、验收
README / 模板 / 脚本 / 验证记录
```

如果这些词出现在前半段，不能解释为“只是顺带提到”，必须视为技术前置失败，重写前半段。

前半段需要表达“能讲产品设计思路、技术架构、技术细节”这类用户补采材料时，必须先翻译成读者语言：

```text
产品设计思路 -> 这个产品为什么要这样解决问题
技术架构 -> 这套判断流程为什么能反复用
技术细节 -> 哪些设计能证明它不是一句口号
```

但翻译后的内容也只能停留在价值判断层，不得提前展开具体架构、模块、文件和验证方式。

技术段标题和过渡句也必须是对外正文，不能暴露作者/模型的结构安排。

禁止使用这类标题：

```text
给懂行的人看的技术背书
技术背书
技术细节说明
后面的技术细节
```

允许改成读者能自然理解的问题式标题：

```text
它为什么不是一句提示词
这套系统怎么把项目接住
为什么它能反复用
它怎么避免把材料写乱
```

禁止使用这类元叙事过渡句：

```text
如果只看到这里，普通读者已经能理解这个产品解决什么问题。
后面的技术细节，只是用来证明它不是一个普通写稿提示词。
给懂行的人看。
普通读者看到这里就够了。
下面进入技术背书。
```

这类话只能作为后台写作判断，不能出现在公开稿。公开稿需要进入技术段时，直接承接前文问题，讲具体设计如何解决具体风险。

技术段必须讲具体机制，不能只写抽象背书。至少选择 3 个可公开机制展开：

```text
薄母技能只做入口分类、状态检查和路由，避免把所有逻辑塞进一个提示词。
固定阶段状态让系统知道现在做到哪一步，避免每轮重新问“下一步是什么”。
运行目录和索引把材料、复盘、输出、反馈分开保存，避免聊天上下文污染公开稿。
内容拆解前先做实践复盘，避免 raw material 直接变成文章。
反馈归因支持 URL 和截图两种入口，把读者反馈接回下一轮行动。
经验收口把重复错误写回 skill、模板或记忆规则，避免同类错误反复发生。
HTML 主交付物和后台记录分离，避免把验收、路径、run_id 写进公开内容。
```

讲技术机制时，不能只写“做了什么”，还必须写“为什么必须这样做”。尤其是多 skill 系统、母技能、入口路由这类设计，必须解释 LLM 执行约束：

```text
LLM 的注意力和执行目标有限。
入口承担太多任务，会让分类、状态判断、材料理解、正文生成、规则修正互相抢注意力。
入口越重，越容易偏离用户这一轮真正要做的事。
所以母技能必须薄，只负责判断输入、读取状态、路由到对应子技能。
真正的内容生成、复盘、反馈归因和经验写回交给子技能处理。
```

禁止只写：

```text
第一层，是入口不要变重。
母技能只做三件事。
这个拆法很关键。
```

如果写“入口不要变重”，必须紧跟原因：注意力有限、目标容易漂移、任务容易混在一起、薄入口如何降低跑偏概率。

技术内容只能放在后半段，且必须用以下目的承载：

```text
向读者证明：它不是一句口号。
解释为什么这套系统能持续复用。
说明哪些设计支撑了前文提到的判断流程。
```

生成公开稿后必须自检：

```text
技术架构是否出现在文章前 50%？
母技能 / 子技能是否在读者痛点讲清楚前出现？
README、模板、脚本、验收是否被写成正文主线？
技术段是否服务于“可信度背书”，而不是抢走主叙事？
技术背书标题之前是否出现了技术架构、技术细节、母技能/子技能、多技能协作、模块分工、模板、脚本、验证记录等词？
```

任一回答为“是”，必须重写，把技术段后移。

## 信息增量协议

产品、skill、workflow、工具、手册或系统型实践的公开稿，必须先证明“这篇内容值得读”，不能只证明“这个产品存在”。

首版内容至少包含：

```text
1. 尖锐问题或真实冲突：目标读者为什么会卡住。
2. 具体案例入口：从哪一次实践、产品、失败、改版或使用场景切入。
3. 失败诊断：旧做法为什么不行，哪里具体失效。
4. 方法增量：这次实践提供了什么新判断、新拆法、新流程或新标准。
5. 可执行动作：读者看完能照着做什么，或能用什么问题检查自己。
6. 待补实践细节问题：哪些关键经历还缺证据，需要用户补充。
7. 修订路径：用户补充后如何改回同一篇稿，而不是另起炉灶。
```

缺任一项时，不得标记为“可发布稿”，只能标记为“可修订框架稿”。

对 AI Flywheel Builder 这类产品，内容重点不是“它有几个模块”，而是：

```text
它如何判断一个真实项目值不值得讲；
如何把个人经历翻译成公共问题；
如何把材料拆成素材、证据、主张、产物和反馈入口；
如何根据缺失信息继续采访用户并修订同一篇稿。
```

禁止输出只有以下特征的稿件：

- 开头是泛泛的”很多人用 AI 会遇到问题”。
- 中段只是介绍产品由哪些模块组成。
- 结尾只是说”把实践变成内容飞轮”。
- 没有具体失败、具体判断、具体方法或具体下一步。
- 没有告诉用户还要补哪些实践细节。
- 中文正文夹杂英文单词（prompt / workflow / README / evidence 等有中文等价词的）。
- 使用编程场景词汇描述写作场景（”编辑器”代替”文档”）。
- 使用法律/行政用语替代日常表达（”证据”代替”依据”）。
- 把内部架构术语直接写进正文（”母技能””薄母技能””子技能”等）。

## 框架先行协议

当项目材料足以判断文章主线，但缺少真实开发困难、失败方案、解决过程、关键转折或公开边界时，先输出“内容框架稿”，不要伪装成最终可发布稿。

内容框架稿的目标是让用户判断：

```text
这些事是不是该讲；
这个角度是不是对；
哪些地方太泛；
哪些真实经历需要补进去。
```

内容框架稿必须包含：

- 暂定标题
- 核心主张
- 读者为什么关心
- 具体案例入口
- 正文结构
- 每一节要讲的判断
- 当前缺少的真实细节
- 1-5 个最小补采问题
- 用户补充后同稿修订动作

用户确认框架方向后，下一步必须进入补采和改稿，不要重新解释飞轮流程，不要重新生成一份无关稿。

### 转折因果桥规则

公开稿不能用“转折点出现在这里”“关键转折来了”这类标签句硬切段落。转折必须写出读者能理解的因果桥：

```text
原来为什么卡住
换了什么动作
这个动作为什么能改变局面
系统因此看到了什么材料
这些材料支撑了哪些判断
具体实现放到后半段再讲
```

处理产品目录、skill 目录、workflow 目录这类案例时，必须讲清楚：

```text
只给 AI 一段产品介绍，它只能顺着用户的说法润色。
把整个项目目录交给它读，它能同时看到 README、产品简报、架构文档、skill 规则、模板、输出记录等材料。
这些材料不一定原样进入公开稿，但能帮助系统判断产品是什么、为谁设计、解决什么问题、哪些证据能公开、哪些只能留在后台、还缺哪些创作者经历。
```

禁止只写：

```text
转折点出现在这里。
关键转折来了。
从这一刻开始，一切都不同了。
```

如果写“转折”，必须紧跟“为什么这是转折”。否则删除“转折”这个词，直接写具体变化。

公开稿叙事实顺序必须忠于真实运行过程，不能为了让故事更顺，把流程改写成另一件事。

处理“产品目录 -> 内容框架 -> 用户补采 -> 同稿修订”这类案例时，必须按实际顺序写：

```text
1. 用户提交产品目录或项目材料。
2. 系统先从目录提取事实和判断，生成一份完整内容框架初稿。
3. 如果同时生成 HTML 和 Markdown，要说明 HTML 是给人阅读判断的，Markdown 是给后续 AI 修订继续读取的。
4. 用户看完框架稿后，按反馈入口补充使用前状态、关键转折、公开边界等经历层信息。
5. 系统再把补充材料转写进同一篇稿，而不是另起一篇。
```

禁止把上述过程改写成：

```text
系统没有先输出稿子，只是先拆问题。
系统先采访完用户，再生成框架。
系统直接生成宣传稿。
系统一次性就写出了发布稿。
```

如果真实过程是“先出框架，再让用户补采”，公开稿必须保留这个事实。它本身就是产品能力的一部分：先给用户一个可判断的初稿，再让用户补最少的真实经历。

## 公开稿与后台记录隔离

`content-flywheel-run.html` 和 `content-flywheel-run.md` 是用户可见交付物，只能放外部读者能看的内容、必要的补采问题和修订入口。

## 用户决策稿闸门

主交付物首先是给用户看的，不是给 AI、作者、编辑或验收脚本看的。

生成 `content-flywheel-run.html` 和 `content-flywheel-run.md` 前，必须先回答：

```text
用户打开这份文件以后，能立刻做什么判断？
用户是否能看懂推荐结论，而不是看见一堆过程字段？
这份文件是在帮用户决策，还是在展示 AI 怎么工作？
哪些内容只应该留在后台，不应该进入主交付物？
```

如果主交付物不能让用户直接判断以下问题，就必须重写：

```text
这件产品/项目应该怎么对外讲？
推荐先讲给谁？
为什么这么讲？
现有材料能支撑什么？
现在还缺哪几句话或哪段真实经历？
用户下一步只需要判断什么？
```

主交付物禁止以后台字段、流程字段或验收字段作为主要结构。以下内容只能放在 `content-package.md`、`learning-closure.md`、索引文件或最终回复的路径清单里，不能进入主交付物正文：

```text
交付物清单
绝对路径表
验收结果
已更新文件
后台自检表
run_id
下一步路由字段
```

最终回复仍必须按 Delivery Tail Rule 列出交付物清单和绝对路径；但这些路径清单不应污染 HTML / Markdown 主交付稿本体。

## 反馈引导闭环

用户看完主交付物后，不能只收到“建议你先看”，还必须知道该怎么反馈。

`content-flywheel-run.html` 和 `content-flywheel-run.md` 的末尾必须包含一个用户可直接回复的反馈入口，回答以下问题：

```text
用户现在只需要反馈哪 1-3 件事？
用户可以用什么格式直接回复？
如果用户认可，下一步会发生什么？
如果用户否定，优先改哪一个判断？
```

反馈入口必须短、具体、可复制。不要让用户重新组织材料、填长表或猜反馈格式。

推荐格式：

```text
1. 首发读者：对 / 不对。应该改成：
2. 主线：对 / 不对。最该强调的是：
3. 案例：就用当前案例 / 换成别的案例。原因是：
```

如果用户确认方向，不得跳过实践细节补采直接扩写终稿。方向确认只代表“框架成立”，下一步必须先补最小必要经历：

```text
1. 使用前后对比：没有这个产品/流程之前具体卡在哪里，用完之后多出了什么判断或动作？
2. 关键转折：开发或使用过程中哪一步让事情真正跑起来？
3. 公开边界：哪些细节能公开讲，哪些只能作为后台证据？
```

只有补完上述经历，才能把可修订稿扩成发布稿。否则只能继续标记为可修订稿。

注意：这里说的是“框架稿已经交付以后”的补采，不是产品目录扫描后的前置提问。不得把这条理解成“先补采再出框架”。

如果用户已经说“我看完了”，下一次回复必须先引导反馈，不能只解释文件改了什么。

最终对话回复也必须包含反馈引导。不能只把反馈问题写在 HTML / Markdown 文件里。

当交付的是可修订稿、内容决策稿、框架稿、首发稿或产品发布文时，最终回复在“交付物清单与绝对路径”之前，必须直接写出用户现在要回答的 1-3 个问题，并说明：

```text
用户怎么直接回复；
如果用户认可，下一步怎么接；
如果用户否定，优先改哪个判断。
```

这样下一轮上下文才能直接接上用户反馈，而不是重新解释文件。

公开稿也不能混入给 AI、作者或编辑看的写作指令。以下表达只能出现在后台拆解、自检或 skill 规则里，不能出现在 HTML 主交付物、Markdown 主交付物或可发布正文里：

```text
要讲的判断
这一节要避免
这一节的信息增量
不要写成
讲清楚
要展示
用户补充后同稿修订动作
同稿修订动作
```

公开稿里如果需要保留框架，只能写读者能直接理解的内容结构，例如：

```text
这一节读者会看到什么问题。
这一节会用哪个案例说明。
这一节能让读者拿走什么判断或动作。
下一版还需要补哪些真实细节。
```

判断标准：如果一句话是在指挥作者怎么写，而不是在给用户看内容本身，就必须移到 `content-package.md` 或 skill 规则里，不能放进主交付稿。

以下内容只能写入后台文件，不得进入公开稿正文、HTML 主交付物或可发布内容草稿：

```text
旧输出作废
本轮重扫
run / run_id / run_uuid
content-package
learning-closure
material-index / output-index / learning-index
修改了什么 skill
验证脚本
验收结论
后台记录
已更新文件
```

如果需要向用户说明后台文件位置，只能放在最终回复的“交付物清单与绝对路径”里，不能写进公开稿本体。

## 判断优先规则

内容拆解的核心不是先展示“系统输出了什么”，而是先展示“系统如何判断什么值得输出”。

处理 AI Flywheel Builder 或类似 workflow / skill / tool 产品时，必须优先展开以下判断机制：

1. 这个项目值不值得讲。
2. 它应该讲给谁。
3. 它能从个人经历翻译成什么公共问题。
4. 核心主张是什么。
5. 哪些只是素材，哪些是证据。
6. 哪些部分能变成可复用产物。
7. 哪些内部过程、私聊上下文、调试细节不能进入公开稿。
8. 哪些反馈应该进入下一轮实践。

内部机制、文件路径、状态字段和验收规则只能服务于上述判断，不能反过来成为文章主线。

## 实践细节补采

公开稿必须包含真实项目的关键经历，尤其是开发或执行过程中遇到的困难、转折和解决方法。只看项目目录、README、skill 文件和后台记录，通常不足以写出有血肉的稿子。

## 补采材料转写规则

用户补充的使用前状态、转折点、公开边界、开发细节、失败过程和判断变化，是写作素材，不是正文原文。

处理补采材料时，必须先写入规则和后台判断，再生成主交付稿。不能直接把用户聊天内容、补采答案或“能讲 / 不能讲”清单贴进公开稿正文。

生成顺序必须是：

```text
1. 先把用户补充归类为：使用前状态 / 转折点 / 可公开信息 / 不公开边界 / 可写成正文的判断。
2. 再判断这些材料应该进入正文的哪一段：痛点、转折、方法、技术背书、边界处理或反馈入口。
3. 把材料转写成读者能读的叙事、解释、判断或例子。
4. 删除后台清单口吻。
5. 再生成 HTML / Markdown 主交付稿。
```

禁止把以下内容直接放进公开主交付稿：

```text
这篇可以公开讲：
这篇不应该公开讲：
能讲的是：
不能讲的是：
可公开边界：
用户说：
用户补充：
内部口径等等乱七八糟的开发口径
```

正确写法：

```text
用户说“能讲产品设计思路” -> 正文里展开：为什么这样设计，解决了什么判断问题。
用户说“能讲技术架构” -> 正文里展开：架构如何支撑这个判断流程。
用户说“能讲技术细节” -> 正文里展开：哪些细节证明它不是一句口号。
用户说“不能讲内部口径” -> 正文里体现为取舍：公开稿只保留对外读者能理解的设计、架构和方法，把临时沟通、后台修正和上下文噪音留在系统内部。
```

如果用户明确指出“先改 skill，再按 skill 写”，必须先完成 `SKILL.md` / 模板 / 记忆规则更新，再改主交付物。不能先改稿再补规则。

用户解释“为什么要重做、为什么旧输出错、为什么重新扫描、为什么不满意”时，默认这是内部修正依据，不是公开内容素材。除非用户明确说“把这段写进正文/公开讲这个过程”，否则不得把这类解释写进首发稿、HTML 主交付物或对外产品叙事。

处理规则：

- 作为后台复盘、经验收口或规则修正证据使用。
- 可以用来判断旧输出哪里错、哪些规则要改。
- 不得自动写成公开故事、产品摩擦或发布稿主线。
- 如果这段经历确实可能有公开价值，先问用户是否允许公开，而不是默认写进去。

如果材料里缺少以下信息，不能假装已经知道；必须在交付稿里标出待补位置，并向用户提出 1-5 个最小必要问题：

```text
最开始为什么要做这个实践？
做的时候最卡的地方是什么？
你试过哪些方案，哪些失败了？
最后是怎么解决的？
解决之后产生了什么变化？
这次实践让你改变了哪个判断？
哪些细节能公开，哪些不能公开？
```

追问规则：

- 不让用户填长表。
- 每轮最多问 1-5 个最关键问题。
- 能从材料中推断的先推断，并标注“这是推断，请用户纠正”。
- 不因为缺少细节而停止出稿；产品目录事实层足够时，先生成可修订草稿，同时明确哪些段落需要用户补充。
- 用户补充后，必须继续修订原稿，而不是重新解释流程或另起一份无关稿。

## 发布稿自检

首发稿写完后，必须先自检，未通过不得标记为可发布稿：

```text
是否具备信息增量，而不是只有顺滑的产品介绍？
是否包含尖锐问题、具体案例入口、失败诊断、方法增量和可执行动作？
如果缺少关键实践细节，是否明确标记为可修订框架稿，而不是可发布稿？
公开稿正文是否完全隔离了 run、索引、验收、旧输出作废、后台记录和已更新文件？
前 30% 是否能让普通目标用户看懂？
是否先讲判断价值，而不是先讲系统结构？
是否把技术细节放在后半段作为可信度证据？
是否把内部术语翻译成人话，或移到附录？
是否删除了用户私聊、情绪表达、调试过程和上下文污染？
是否围绕真实项目 -> 判断 -> 内容 -> 产物 -> 反馈 -> 下一轮行动展开？
是否包含开发/执行过程中的真实困难、尝试、解决和转折？
是否按产品发布长文协议组织，而不是说明书或功能列表？
是否明确这是多 skill 协作系统，而不是 prompt 或单个写作 skill？
```

## 外部语言闸门

公开稿必须使用目标读者会说、会搜、会转述的语言。内部口径、自造词、项目内黑话、临时命名和非行业通用术语，默认不得进入公开稿正文。

## 中文输出风格硬规则

面向中文用户输出时，必须使用简体中文。

**唯一判断标准：咖啡测试。** 交付前，把整篇稿子在心里用中文念一遍。任何一句话，你跟朋友喝咖啡的时候不会这么说——重写。不需要判断”这个词在不在禁词列表里”，只需要判断”人会不会这么说”。

三类最常见的失败模式：

```text
1. 英文直译腔：脑子里跑英文，硬翻成中文。
   “通过对项目目录进行扫描，以实现内容框架的生成” → 不是人话。
   “系统先读完整项目目录，再给出一版内容框架” → 是人话。

2. 内部文档腔：把系统架构术语当成读者能懂的话。
   “母技能只做三件事” → 读者不知道什么是母技能。
   “系统入口只做三件事” → 读者能懂。

3. 书面语过度：用了只存在于书面/学术/律法场景的词。
   “证据” → 行政律法用词，正常人说”依据”。
   “归因” → 数据分析术语，正常人说”搞清楚问题出在哪”。
```

有公认中文等价词的英文，用中文：提示词（不是prompt）、工作流（不是workflow）、说明文档（不是README）。

没有公认中文等价词的专有名词保留英文：LLM、HTML、Markdown、API。

生成用户可见内容前，做一轮咖啡测试：

```text
1. 念一遍。这句话像不像翻译软件翻的？
2. 像不像内部文档里才有的词？
3. 像不像只在书面/学术/律法场景才用的词？
4. 能不能用更短、更直接的方式说？
5. 如果像，重写。不要查词表，靠耳朵判断。
```

## 人话改写硬闸门

公开稿、HTML 主交付物、Markdown 主交付物、内容框架稿在交付前，必须先做一轮”人话改写”，不能等用户指出后才改。

**不要用词表。用咖啡测试。** 词表永远追不上模型生成变体的速度。”薄母技能”改掉了会有”硬设计约束”，改掉了会有”判断成本”。唯一的判断标准是：念一遍，你跟朋友喝咖啡的时候会不会这么说。

生成顺序：

```text
1. 先写出产品事实和内容判断。
2. 从头念一遍。耳朵找三类问题：
   a. 这个词是从英文硬翻过来的吗？（prompt→提示词、evidence→依据）
   b. 这个词是内部文档/架构术语吗？（母技能→入口层、经验收口→写回规则）
   c. 这个词是法律/学术/律法书面语吗？（证据→依据、归因→搞清楚）
3. 标记每一处不像人话的地方。
4. 用自然中文重写。不看词表，靠耳朵判断。
5. 再念一遍。确认没有漏网。
```

咖啡测试自检：

```text
这句话像不像真实的人会说的话？
第一次看到这个产品的人能不能马上懂？
它是不是只在系统内部成立，对外读者不关心？
读者看到后会不会问：”这到底是什么东西？”
```

任一问题回答为”否 / 会”，必须重写，不能交付。

**不是绝对禁用某类词，而是判断”读者会不会这么说”。** 如果一个词在读者日常词汇里不存在——无论它在内部文档里用了多少次——它就不能进入公开稿。不需要查禁词列表，耳朵会告诉你。

以下词默认属于内部口径，不能直接出现在公开稿前半段：

- 主交付物
- 经验收口
- 状态流转
- 模块路由
- 后台记录
- run_id
- output_id
- material-index
- learning-index
- flywheel-state
- current_stage
- active_module
- missing_requirements
- 验收脚本
- 内容包
- 发布前内容判断卡

允许替换成人话：

```text
主交付物 -> 跑完以后能直接拿去用的结果
经验收口 -> 把这次踩坑写成下次会遵守的规则
状态流转 -> 系统知道现在做到哪一步
模块路由 -> 系统知道下一步该调用哪种能力
索引/后台记录 -> 系统把材料整理好，方便下次继续用
发布前内容判断卡 -> 发布前先判断这件事讲给谁、讲什么、哪些不能讲
```

如果一个词既不是目标读者常用词，也不是业界通用词，也没有在正文里用一句人话解释，就必须删除或替换。不要为了显得系统复杂而创造术语。

## 失败模式

- 不要替用户默认写完整手册本体。
- 不要把内容角度列表当成最终交付。
- 不要只有后台文件而没有用户可见主交付物。
- 不要只生成 Markdown 主交付物；面向人类阅读的主交付物优先生成 HTML，Markdown 只作为后台源码版或可编辑草稿。
- 不要把私有设计 skill 当作 HTML 生成的硬依赖；没有指定主题时使用默认 HTML 主题。
- 不要把“产品理解”和“发布前内容判断卡”拆成两轮重复确认。
- 不要把内部验收、目录、脚本、索引、状态流转、运行记录写成对外内容主线。
- 不要把技术细节、状态机、中控、索引、验收脚本放在对外稿开头；它们只能在后半段做可信度背书。
- 不要把用户私聊、情绪表达、调试过程或纠错对话写进公开稿。
- 不要把用户解释旧输出为什么错、为什么重扫或为什么不满意的内部说明，默认写进公开稿。
- 不要把“交付物格式”当成产品叙事中心；它只是防止用户空手的质量底线。
- 不要在公开稿里使用自造术语、内部口径或项目黑话；如果读者不会这么说，就先翻译成人话。
- 不要在对象已处理时只提示提交反馈，而不返回已有处理结果摘要。
- 不要写没有证据锚点的发布稿。
- 不要写没有事实锚定表的公开稿；尤其不要把过程顺序、用户动作、系统动作、交付物形态和补采过程凭感觉写成事实。
- 不要在产品目录事实层足够时先问用户问题；必须先交完整框架稿和 HTML/Markdown 双交付，再在稿末引导补采。
- 不要在缺少开发困难、解决过程和关键转折时，把稿子伪装成最终版。
- 不要把交付稿当一次性工作；用户补充细节后必须修订同一稿件。
- 不要把产品、skill 或 workflow 的发布稿写成说明书、功能列表、README 复述或验收报告。
- 不要把 AI Flywheel Builder 降级描述成 prompt 或单个写作 skill；它是一套多 skill 协作的内容创作飞轮。
- 不要把顺滑但泛泛的产品介绍当成可发布内容；没有信息增量就只能作为可修订框架稿。
- 不要在缺少具体案例、失败诊断、方法增量和可执行动作时标记为可发布稿。
- 不要把 run、索引、验收、旧输出作废、后台记录、已更新文件写进用户可见主交付物。
- 不要忽略平台和公开边界风险。
- 不要在内容包完成后停在“你可以选择”。
- 不要在未做经验收口时结束本模块。
- 不要只说生成了文件；必须在回复末尾列出交付物绝对路径，并引导用户先看哪些内容。
- 不要只把反馈入口写进文件；最终对话回复本身也必须问清楚用户该反馈什么。
