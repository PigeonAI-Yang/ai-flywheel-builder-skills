# Product Brief

## Product Name

AI Flywheel Builder

## One-liner

帮助用户建立个性化 AI 内容创作飞轮：先通过初始化建立用户画像、内容种子库、内容主线、产物方向和反馈机制，再由后续模块设计实践并把实践持续转化成内容、产物、反馈和下一轮行动。

## Long-term Direction

当前版本从 AI 实践型创作者场景切入，因为这是第一批真实验证场景。

长期目标不是只服务 AI 创作者，而是做成通用内容创作飞轮：

```text
任何有真实实践的人 -> 提交实践 / 产品 / 项目 / 作品 / 经验 -> 系统判断什么值得讲 -> 生成内容框架 -> 补采真实经历 -> 形成可发布内容、可复用产物、反馈入口和下一轮行动
```

通用版本不能只处理产品目录。它必须支持四类内容源头：

```text
实践输入：我做了什么、服务了谁、完成了什么项目、踩了什么坑。
产品输入：我做了一个产品、工具、skill、手册、模板、工作流或作品。
观察输入：我看到了什么现象、行业变化、用户行为或反复出现的问题。
观点输入：我形成了什么判断、认知变化、反常识观点或争议性问题。
```

后续版本需要逐步把“AI 创作者”抽象成更通用的“实践型创作者”。适用对象可以扩展到：

- 独立开发者
- 知识工作者
- 顾问
- 设计师
- 运营
- 教育者
- 手艺人
- 小团队主理人
- 任何持续做事、复盘、输出的人

抽象时不能丢掉当前已经验证有效的核心：

- 真实实践是内容源头
- 系统负责判断什么值得讲
- 内容要沉淀成可复用产物
- 反馈要回到下一轮实践
- 用户不负责推进流程，AI 负责推进流程

## Core User

想做 AI 实践型创作者、独立开发者、顾问或小团队主理人的用户。他们经常不知道创作什么，不知道自己有哪些实践值得讲，也不知道下一步可以做什么实践来产生内容。

## Pain

- 不知道自己可以创作什么。
- 不知道已有经历、AI 使用方式、失败、问题或长期项目里哪些值得讲。
- 不知道下一步可以做什么小实践，才能产生真实内容。
- 即使做了实践，也难拆成可传播的主题、脚本和产物。
- 内容选题依赖灵感，不能稳定从实践里长出来。
- 内容、产物、用户反馈和下一轮实践没有形成闭环。

## Product Promise

通过母 skill 进行必要访谈，生成个性化飞轮蓝图，用户确认一次后，由 AI 批量初始化长期记忆、产物登记表、当前焦点、反馈机制和子 skill 激活方案。

核心循环是：

```text
初始化：用户画像 -> 内容种子库 -> 内容主线 -> 产物方向 -> 反馈机制 -> 记忆更新规则
运行：实践设计 -> 实践过程 -> 发现/摩擦 -> 内容主题 -> 内容草稿/脚本 -> 可领取产物 -> 用户反馈 -> 下一轮实践
```

初始化完成后，如果用户还不知道做什么，系统进入实践设计模块，生成实践池和第一轮实践建议。如果用户已经有真实实践对象，例如目录、repo、对话、实验、工作流、失败案例或反馈样本，系统继续完成：

```text
用户资料或实践对象 -> 内容种子 -> 可做实践 -> 内容主题 -> 内容草稿/脚本 -> 可领取产物 -> 反馈回流 -> 下一轮实践动作 -> 后台记录
```

成果处理只是其中一个子模块。当用户提交已有 app、skill、手册草稿、工作流、产品目录或原型时，系统会把它转成内容素材、可领取产物和下一轮实践问题，而不是把产品定位成只处理已有成果的工具。

## Modules

### Module 1: 初始化

适用对象：

```text
新用户第一次使用，还没有个人内容创作飞轮。
```

目标：

```text
分层采访 -> 用户建模 -> 生成个性化飞轮蓝图 -> 用户确认 -> 批量初始化飞轮系统
```

初始化阶段不生成公开内容、不生成自检清单、不要求用户先提交实践。它必须先建立：

- 用户画像
- 内容种子库
- 内容主线
- 产物方向
- 反馈机制
- 记忆更新规则
- 当前焦点
- 下一步自动动作

### Module 2: 实践设计

初始化完成后，如果用户不知道做什么，系统从用户画像、内容种子库和内容主线里设计下一轮小实践。

实践设计模块负责生成实践池和第一轮实践建议。

### Module 3: 实践处理

用户提交实践后，系统先接收材料、建立运行编号、归档资料并更新索引，然后才进入复盘、内容主题、内容草稿、可领取产物、反馈回流和下一轮实践。

用户允许乱提交材料。整理资料、分类和索引是 AI 的职责。

实践复盘模块负责判断本轮实践质量，提取摩擦、发现、内容种子、产物机会和长期记忆候选，并给出下一步路由。用户不需要自己判断材料是否值得讲。

内容拆解模块负责把复盘后的真实实践或已有产物拆成内容包，并产出首发稿、产物发布说明、证据清单和反馈入口。它不默认写完整产物本体；如果产物本体未成型，先路由到产物设计或构建模块。

## Target Experience

AI Flywheel Builder 的入口体验应接近 Nuwa skill：

```text
必要采访 -> 生成飞轮蓝图 -> 用户一次确认 -> AI 批量完成初始化 -> 交付可运行系统
```

它不是让用户持续回答“下一步做什么”的建议器，也不是把流程拆成很多用户任务的清单工具。

## 固定输出

V0.1 的初始化文件包括：

- 用户飞轮画像
- 产物登记表
- current focus
- activation plan
- friction log
- feedback log
- practice submission
- reverse validation report

初始化运行的主交付物包括：

- 用户画像
- 内容种子库
- 内容主线
- 产物方向
- 反馈机制
- 记忆更新规则
- 当前焦点
- 下一步自动动作

实践处理运行的主交付物包括：

- 一个用户可见 HTML 主交付文件：`J:/pigeonAI/AI-flywheel-Builder/workspace/{user}/outputs/{run_id}/content-flywheel-run.html`
- 一个 Markdown 源文件或可编辑草稿：`J:/pigeonAI/AI-flywheel-Builder/workspace/{user}/outputs/{run_id}/content-flywheel-run.md`
- 内容种子
- 实践复盘
- 第一轮最小实践
- 本轮内容主题
- 若干内容选题
- 一篇内容草稿或短视频脚本
- 一个配套可领取产物
- 反馈回流方式
- 下一轮实践动作
- 后台更新摘要

如果提交的是已有 skill、app、手册或工作流，可以额外生成产品定位、发布前验收清单、使用说明草案和下一版体验问题清单。

其中 `J:/pigeonAI/AI-flywheel-Builder/templates/practice-submission.md` 和 `J:/pigeonAI/AI-flywheel-Builder/templates/reverse-validation-report.md` 是真实实践运行后的收口组件。没有验收结论，就不算完成一次飞轮运行。

但收口组件不是主交付物。初始化主交付物必须先建立用户飞轮系统；实践设计和实践处理模块再帮助用户知道该做什么实践，以及如何形成内容、产物、反馈和下一轮实践。

面向用户阅读、发布和展示的主交付物优先使用 HTML。Markdown 只作为后台源码版、可编辑草稿或同步记录。

HTML 默认使用开源内置主题：

`J:/pigeonAI/AI-flywheel-Builder/themes/default-html-report/`

如果用户明确指定本地设计 skill 或主题，可以使用可选主题。PigeonAI / 雾灯会设计范式只属于作者本机可选主题，不进入开源核心依赖。可选主题不可用时，系统必须回退到默认主题。

实践处理运行的主交付物最低包含：

- 本轮实践一句话
- 为什么值得讲
- 可发布内容草稿
- 配套可领取产物
- 证据与边界
- 反馈入口
- 下一轮实践

## 路径规则

系统向用户报告文件、模板、工作区产物或下一步执行对象时，必须提供绝对路径。

原因：相对路径会让用户额外判断“这是哪里的文件”，增加认知负担。绝对路径直接对应用户机器上的真实对象。

## Operating Principle

AI Flywheel Builder 不是建议器，而是个人 AI 内容创作飞轮初始化器和运行器。

用户不应该被迫思考：

- 下一步做什么
- 调用哪个子 skill
- 如何测试
- 写到哪个记忆文件
- 当前是否跑偏

这些由系统负责。

用户只需要确认、选择和补充真实信息。

当用户说“不知道下一步做什么”或“推进很吃力”时，这不是用户能力问题，而是产品失败信号。初始化阶段必须回到“访谈-蓝图-确认-批量执行”协议；初始化完成后，系统用单线程工作包维护推进。

## V0.1 Scope

V0.1 只定义产品架构和第一组 skill 草案：

- 薄母 skill：`ai-flywheel-builder`
- 初始化模块：`flywheel-initializer`
- 实践设计模块：`practice-designer`
- 实践接收模块：`practice-capture`
- 实践复盘模块：`practice-reviewer`
- 中控编排：`flywheel-orchestrator`
- 长期记忆：`flywheel-memory`
- 摩擦采集：`friction-capture`
- 产物设计：`artifact-designer`
- 内容拆解：`content-decomposer`
- 反馈归因：`feedback-attributor`
- 飞轮守门：`flywheel-gatekeeper`

不做：

- Web app
- 自动化 UI
- 多用户数据系统
- 发布平台集成
- 让用户手动编排子 skill
