# 路线图与版本规划

## 作用

本文件统一记录 AI Flywheel Builder 的后续版本规划、已裁定但暂不实现的能力、以及从当前版本到成熟形态的演进路径。

规则：

- 已完成变化写入 `J:/pigeonAI/AI-flywheel-Builder/CHANGELOG.md`。
- 发版规则写入 `J:/pigeonAI/AI-flywheel-Builder/docs/02_VERSIONING.md`。
- 未来规划和暂缓能力写入本文件。
- 用户已经裁定但本轮不实现的能力，必须记录到本文件，不能只停留在聊天里。

## 当前版本

`0.1.0`

目标：

```text
先完成可运行的内容创作飞轮链路：
初始化 -> 实践设计 -> 实践接收 -> 实践复盘 -> 内容包 -> 反馈归因 -> 下一轮实践
```

## 版本路线

### v0.1：单次飞轮运行可用

目标：

- 完成母 skill 路由。
- 完成初始化模块。
- 完成实践设计模块。
- 完成实践接收模块。
- 完成实践复盘模块。
- 完成内容拆解模块。
- 完成反馈归因模块。
- 建立按 `run_id` 管理的工作区、索引和模板。
- 固定中控调度、状态枚举、用户可见主交付物和 V0.1 验收脚本。

验收：

```text
用户可以从“不知道做什么”开始，完成一次实践，并得到用户可见主交付物、内容包、反馈入口、经验收口和下一轮实践方向。
```

硬标准：

```text
1. 用户不用追问“然后呢”。
2. 有一个绝对路径的主交付物。
3. run 目录、索引、经验收口全部闭环。
```

### v0.2：通用内容创作飞轮能力

目标：

- 保留 v0.1 已验证的 AI 实践型创作者场景。
- 将产品定位从“AI 内容创作飞轮”升级为“通用内容创作飞轮”。
- 把用户对象从 AI 创作者扩展为所有实践型创作者。
- 把输入对象从 AI 产品、skill、workflow 扩展为更通用的实践、项目、作品、服务、案例、实验、失败、经验、观察、观点和判断。
- 让初始化、实践设计、实践接收、实践复盘、内容拆解、反馈归因都支持非 AI 场景。
- 当用户不知道选题、不知道可以创作什么时，系统不能只等用户提交产品；必须通过采访、经历盘点、观察捕捉和小实践设计，主动生成内容方向。
- 当用户提交一个观点或判断，例如“只要还是 Transformer 路线，AI 很难走向 AGI”，系统要按观点型内容处理：澄清概念、拆出论证结构、列证据缺口、整理反方观点、设计验证动作，而不是要求用户必须先有产品。

核心抽象：

```text
AI 创作者 -> 实践型创作者
AI 实践 -> 真实实践
AI 产品 / skill / workflow -> 产品 / 项目 / 作品 / 方法 / 服务 / 案例 / 实验
AI 内容创作飞轮 -> 内容创作飞轮
```

v0.2 必须支持四类输入入口：

```text
1. 实践输入：我做了什么、服务了谁、完成了什么项目、踩了什么坑。
2. 产品输入：我做了一个产品、工具、skill、手册、模板、工作流或作品。
3. 观察输入：我看到了什么现象、行业变化、用户行为或反复出现的问题。
4. 观点输入：我形成了什么判断、认知变化、反常识观点或争议性问题。
```

四类输入都必须进入同一个飞轮：

```text
输入 -> 澄清 -> 判断价值 -> 证据/经历/案例补采 -> 内容框架 -> 可发布内容 -> 可复用产物或方法 -> 反馈 -> 下一轮行动
```

不能改变的核心：

- 真实实践是内容源头。
- 真实观察和真实思考也可以成为内容源头，但必须补证据、案例或验证动作，不能写成空泛观点。
- 系统负责判断什么值得讲、讲给谁、用什么证据讲。
- 先产出内容框架，再补采真实经历。
- 内容要转成可发布稿、可复用产物、反馈入口和下一轮行动。
- 用户不负责推进流程，AI 负责推进流程。

验收：

```text
用户提交一个非 AI 实践对象，例如设计作品、咨询案例、运营实验、课程手册、手工项目或小团队服务流程，系统仍能完成：
材料接收 -> 价值判断 -> 内容框架 -> 经历补采 -> 发布稿 -> 可复用产物 -> 反馈入口 -> 下一轮实践。
```

```text
用户只提交一个非产品型观点、观察或困惑，例如“不知道写什么”“我最近觉得某个行业判断变了”“我认为某条技术路线走不通”，系统也能完成：
概念澄清 -> 观点拆解 -> 证据缺口 -> 反方观点 -> 内容框架 -> 补采问题 -> 验证动作 -> 可发布观点稿或暂缓发布理由。
```

### v0.3：平台反馈采集适配器

目标：

- 建立平台采集适配器库。
- `feedback-attributor` 不内置平台细节，只按需加载对应平台采集策略。
- 支持常见平台的只读采集策略。

计划目录：

```text
J:/pigeonAI/AI-flywheel-Builder/references/feedback-collectors/
  generic-web.md
  xiaohongshu.md
  douyin.md
  bilibili.md
  wechat.md
```

每个平台适配器记录：

- 适用页面
- 采集目标
- 操作步骤
- 如何展开更多评论
- 如何识别数据指标
- 失败降级方式
- 禁止操作
- 输出字段

验收：

```text
用户给一个平台反馈页 URL，系统能读取对应 collector，按平台策略只读采集反馈；失败时自动降级到截图。
```

### v0.3：主题系统增强

目标：

- 将 HTML 主交付物主题从“说明文档 + 模板”升级为可选择、可验证的主题系统。
- 支持用户在工作区声明默认主题。
- 支持主题级截图验收。
- 支持把私有品牌主题作为外部插件接入，而不是复制进开源核心。

当前 v0.1 已完成：

```text
J:/pigeonAI/AI-flywheel-Builder/themes/default-html-report/
J:/pigeonAI/AI-flywheel-Builder/themes/pigeonai-ff-optional/
```

暂缓原因：

- 需要更明确的主题 manifest。
- 需要 HTML 渲染截图验收。
- 需要决定主题选择状态写在用户工作区还是项目级配置。

### v0.4：账号登记与按需反馈追踪

目标：

- 用户登记自己的内容账号和后台入口。
- 系统在用户要求下，对已登记账号做只读反馈采集。
- 不保存账号密码，只复用用户本机登录态。

计划模板：

```text
J:/pigeonAI/AI-flywheel-Builder/templates/account-index.md
```

账号登记字段：

- 平台
- 账号名
- 主页 URL
- 后台入口 URL
- 主要内容类型
- 采集权限：只读
- 是否启用周期追踪

验收：

```text
用户说“帮我看一下小红书这个账号最近反馈怎么样”，系统能读取账号索引、打开对应入口、采集可见反馈、更新反馈索引并生成归因报告。
```

### v0.5：周期性反馈监控

目标：

- 支持每日/每周周期采集。
- 对比上次反馈变化。
- 发现异常提醒。
- 汇总高频反馈。
- 将反馈转成下一轮实践建议。

暂缓原因：

- 需要定时任务。
- 需要稳定浏览器登录态。
- 需要平台限制和错误恢复策略。
- 需要更严格的数据量管理。

验收：

```text
系统可以在用户授权下周期性追踪已登记账号，只读采集反馈，并把异常或机会推进到飞轮下一步。
```

## 已裁定但暂不实现

### product self-feedback loop

长期规则：

```text
用户对 AI Flywheel Builder 本身的使用反馈，也必须进入飞轮。
```

处理方式：

- 归因到模块规则缺陷、平台适配缺陷、模板字段缺陷、状态流转缺陷、文档说明缺陷、工具能力限制或路线图候选能力。
- 能立即修正的，写入对应 `skills/*/SKILL.md`、`templates/*.md`、README 或 docs。
- 暂不实现但有价值的，记录到本路线图。
- 不能只停留在聊天解释。

示例：

```text
微信公众号文章反馈页截图失败 -> 先归因是平台适配问题、浏览器能力限制还是证据留存模板缺陷 -> 决定修改 feedback-attributor、feedback-report 模板、wechat collector 规划或路线图。
```

### feedback-monitor

长期方向：

```text
feedback-attributor 从单次 URL/截图反馈归因，升级为用户授权下的多账号只读反馈追踪。
```

边界：

- 只读采集。
- 不保存账号密码。
- 不发布内容。
- 不删除内容。
- 不回复评论或私信。
- 不点赞、关注、收藏。
- 不修改账号或平台设置。
- 不绕过平台权限。

当前状态：

```text
记录为 v0.3-v0.5 规划，不进入 v0.1 / v0.2 主目标。
```
